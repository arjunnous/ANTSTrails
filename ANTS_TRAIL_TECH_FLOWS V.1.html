profile_key derivation + roadmap engine
// Step 1 — derive profile_key from form submission
function deriveProfileKey(primaryTool, experienceLevel) {
  const toolMap = { 'Tosca':'tosca', 'TestComplete':'testcomplete', 'JMeter':'perf',
                    'Postman':'api', 'Manual QA':'manual', 'Playwright':'playwright' };
  const lvlMap  = { 'Junior':'junior', 'Mid':'mid', 'Senior':'senior', 'Architect':'architect' };
  return `${toolMap[primaryTool] ?? 'unknown'}_${lvlMap[experienceLevel] ?? 'mid'}`;
}

// Step 2 — lookup + sort (mandatory first, then role-specific)
function getRoadmap(profileKey, matrix) {
  const profile = matrix.profiles.find(p => p.profile_key === profileKey);
  if (!profile) {
    // Fallback: mandatory courses only + notice
    const all = matrix.profiles.flatMap(p => p.courses);
    return { courses: all.filter(c => c.mandatory).sort((a,b) => a.sequence_order - b.sequence_order), fallback: true };
  }
  const sorted = profile.courses.sort((a,b) => a.sequence_order - b.sequence_order);
  return { courses: [...sorted.filter(c => c.mandatory), ...sorted.filter(c => !c.mandatory)], fallback: false };
}

Progress Tracking R1

Each course has 3 states. Updated via a single click — auto-persisted via UPSERT, no save button. Completion % recalculated and returned inline; the UI patches the DOM without a page reload.

○
Not started
state = 'not_started'
◑
In progress
state = 'in_progress'
✓
Completed
state = 'completed'
PATCH /progress — upsert flow
1
Client request PATCH
PATCH /progress · { course_id, new_state } · Authorization: Bearer <JWT>
2
Auth extracts user_id JWT
req.user.id scopes all DB writes — users can only update their own rows
3
UPSERT progress table SQL
ON CONFLICT (user_id, course_id) DO UPDATE SET state = excluded.state, updated_at = now()
4
Recalculate % COMPUTE
COUNT(*) FILTER (WHERE state='completed') / COUNT(*) * 100 — rounded to nearest int
5
200 inline update 200
Return { state, completion_pct } → UI patches DOM directly, no reload needed
progress table + completion query
CREATE TABLE progress (
  user_id    UUID  REFERENCES users(id) ON DELETE CASCADE,
  course_id  TEXT  NOT NULL,
  state      TEXT  NOT NULL DEFAULT 'not_started',
  updated_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (user_id, course_id)
);

-- Completion % in one query
SELECT ROUND(
  COUNT(*) FILTER (WHERE state = 'completed') * 100.0 / NULLIF(COUNT(*), 0)
) AS completion_pct
FROM progress WHERE user_id = $1;

Course Matrix Content Flow Both

course_matrix.json is the single source of truth for all roadmap logic. Produced by the SME team, consumed read-only by the app. Schema locked at kickoff — changes require all-hands sign-off.

course_matrix.json — locked schema v1.0
{
  "schema_version": "1.0",
  "profiles": [{
    "profile_key":      "tosca_mid",          // join key — must match deriveProfileKey() exactly
    "display_label":    "Tosca Engineer (Mid)",
    "role_tags":        ["automation"],
    "experience_level": "mid",
    "courses": [{
      "course_id":        "AI_BASICS_001",     // globally unique across all profiles
      "title":            "What is AI / Gen AI — Foundations",
      "platform":         "Udemy",
      "url":              null,
      "duration_minutes": 90,
      "mandatory":        true,          // shown to ALL profiles regardless of key match
      "tags":             ["AI basics"],
      "sequence_order":   1              // SMEs own this — app renders ascending
    }]
  }]
}
contract rules
R
profile_key is the join key
Must exactly match deriveProfileKey() output. Lowercase snake_case. Changes require all-hands sync.
R
mandatory: true renders first for all users
No profile key match → mandatory courses only + "custom roadmap coming soon" notice shown to user.
R
App never writes to this file
Read-only at runtime, cached in memory on startup. Content updates: drop new file version → app restart to reload.
R
v2 increments schema_version to "2.0"
Builder handles both "1.0" and "2.0". No breaking changes between versions — additive profile additions only.

// Auth router — feature flag gates between local and SSO
router.get('/login', (req, res) => {
  if (process.env.SSO_ENABLED === 'true') {
    const url = `${process.env.SSO_PROVIDER_URL}?callback=/auth/sso/callback`;
    return res.redirect(302, url);   // → SSO provider (not implemented in R2)
  }
  res.render('login');               // → R1 local form (unchanged)
});

// Callback stub — 501 until IT completes the exchange
router.get('/auth/sso/callback', (req, res) => {
  res.status(501).json({
    error: 'SSO integration pending IT configuration',
    guide: 'docs/SSO_INTEGRATION_GUIDE.md'
  });
});
